> Continuing the "Claude Code Journey" series. Last time we talked about
> [splitting work across subagents]({{ '/en/blog/subagents-ajan-takimlari/' | relative_url }});
> now we look at a more measured way to work within a single session — plan mode.

For some tasks you don't want Claude to start editing files right away. You want
it to understand the codebase first, tell you what it plans to do, and only act
after you approve. **Plan mode** exists for exactly this.

## What does plan mode do?

The official docs put it plainly: in plan mode Claude researches and proposes
changes without making them. It reads files, runs shell commands to explore, and
writes a plan — but it does not edit your source. Permission prompts still apply
just as they do in Manual mode.

So plan mode is a "read only, then explain" mode. Claude walks the project and
presents what it would do as a plan; the code stays untouched.

## How to enter it

There are three ways. Inside a session you cycle through modes with `Shift+Tab`;
the cycle goes `default → acceptEdits → plan`. For a single one-off request you
can prefix your prompt with `/plan`. Or you can start directly in plan mode:

```bash
claude --permission-mode plan
```

If you want to leave plan mode without approving a plan, press `Shift+Tab` again.

Say you're about to overhaul a payments module but want to see Claude's approach
first. While in plan mode you give a request like:

```text
draft a plan to migrate the payments module to Stripe's new API
```

Claude reads the relevant files, examines the existing integration, and hands you
a step-by-step plan — without changing a single line of code.

## Reviewing and approving a plan

When the plan is ready, Claude presents it and asks how you'd like to proceed. At
that point you have options: approve the plan and start directly in auto mode,
approve and accept edits automatically (acceptEdits), approve and review each
edit manually one by one, or give feedback and keep planning.

Approving a plan exits plan mode and switches the session to the permission mode
you chose — so Claude starts editing. If you want to plan again, cycle back to
plan mode with `Shift+Tab`, or prefix your next request with `/plan`.

One small convenience: if you want to edit the plan directly, press `Ctrl+G` to
open it in your default text editor before Claude proceeds.

## Making plan mode the default

If you want plan mode to be the persistent default for a project, use the
`defaultMode` setting in `.claude/settings.json`:

```json
{
  "permissions": {
    "defaultMode": "plan"
  }
}
```

Now every session in that project opens in plan mode; it reads and plans, but
won't touch your files until you allow it.

## When is it worth it?

Plan mode helps most when the direction is uncertain or the change surface is
wide. Before diving into a big refactor, while exploring an unfamiliar codebase,
or ahead of a risky edit when you'd rather say "tell me first." If the plan
looks good, you approve and move to execution; if not, you give feedback and
refine it without a single file changing.

For small, clear tasks it can be an unnecessary step — if what needs doing is
already obvious, working directly is faster. Plan mode is for the cases where you
want to see the decision before it's made.
